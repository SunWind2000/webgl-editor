# 14. 交付阶段、团队与工程组织

## 14.1 阶段与出口条件

| 阶段 | 周期参考 | 主要输出 | 出口条件 |
|---|---:|---|---|
| Phase 0 架构 POC | 4–6 周 | Electron/NestJS 进程模型；WebGPU/WebGL2 适配；OCCT STEP；DWG SDK 双 POC；SAB 调度器原型 | 三项立项决策冻结；样本和基准可重复 |
| Phase 1 MVP | 10–14 周 | 项目、导入任务、glTF/DXF/STEP/PCD、首屏 LOD、选择测量、命令事务、恢复 | P0 格式与核心验收通过 |
| Phase 2 正式 V1 | 12–16 周 | DWG 正式链路、点云八叉树、网格编辑、插件、性能面板、签名更新 | 客户试点数据和安全审计通过 |
| Phase 3 CAD 增强 | 16–24 周 | 精确建模命令、装配/特征、STEP 写出、版本比较 | 精确模型回归与往返策略达标 |

周期按 7–10 人跨职能团队估算，仅用于立项排序；DWG 商业 SDK 采购、客户样本质量和精确建模范围会显著影响计划。

## 14.2 Phase 0 必做 POC

1. Electron 主进程、renderer、NestJS utility process、Native Sidecar 的启动、崩溃和重启链路。
2. WebGPURenderer/WebGL2 统一适配和能力降级。
3. OCCT 读取典型 STEP 装配，输出粗/精 LOD 和拓扑映射。
4. ODA 与 RealDWG 对同一 DWG 样本的保真、性能、包体、授权和平台比较。
5. 100M 点级数据的点云转换、渐进加载和 point budget。
6. Protobuf manifest/chunk、HTTP 二进制、SSE 和本地 CAS。
7. Worker Scheduler、SAB arena、取消、内存准入和 Transferable 降级。
8. 命令日志、稀疏 patch、检查点和崩溃重放。

|---|---:|---|

## 14.4 Monorepo 建议

```text
apps/
  desktop/          # Electron main + preload
  renderer/         # Vue editor + Three engine
  local-service/    # NestJS/Fastify
packages/
  protocol/         # .proto, generated TS/C++ bindings, golden fixtures
  engine-model/     # document/entity/command domain model
  render-core/      # renderer adapter, resources, picking, LOD
  worker-runtime/   # scheduler, task registry, shared arena
  plugin-sdk/       # stable extension API
native/
  geometry-sidecar/ # OCCT + ODA/RealDWG adapters
  common-protocol/
tests/
  fixtures/ perf/ golden/ visual/ security/
```

## 14.5 工程规范

- TypeScript、C++ 和 Protobuf 统一代码生成、格式化、静态检查和版本策略。
- 协议变更必须经过兼容性评审和 golden fixtures 更新。
- 每个格式适配器有独立样本、基准和错误码空间。
- Native Sidecar 使用可复现构建、SBOM、代码签名和多 OS CI。
- 性能回归作为 CI/CD 门禁，不只在版本末期测试。

## 14.6 发布策略

- Windows x64 首发；macOS Apple Silicon 次优先；Linux 根据商业 SDK 支持确定。
- 应用、Sidecar、协议和项目 schema 使用独立版本号。
- 自动更新采用分阶段发布、签名校验、崩溃率监控和可回退版本。
- 项目文件写入新特性前进行能力/版本校验，避免旧版本不可恢复打开。
