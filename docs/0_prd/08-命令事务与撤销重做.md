# 8. 命令、事务与撤销重做

## 8.1 设计原则

- 文档状态由已提交命令产生，Three.js 场景仅是该状态的可丢弃投影。
- 悬停、相机拖拽预览等 UI 临时状态与持久化模型命令分离。
- 一个用户意图可包含多个底层命令，必须作为原子事务提交或回滚。
- 命令包含 `command_id`、`transaction_id`、`document_id`、`base_revision`、优先级和 payload。
- 撤销/重做产生新事务，不删除或重写历史命令。

## 8.2 协议草案

```proto
message CommandEnvelope {
  bytes command_id = 1;
  bytes transaction_id = 2;
  bytes document_id = 3;
  uint64 base_revision = 4;
  int64 created_at_ms = 5;
  CommandPriority priority = 6;

  oneof payload {
    TransformEntities transform = 20;
    DeleteEntities delete_entities = 21;
    UpdateMaterial update_material = 22;
    ApplyMeshPatch mesh_patch = 23;
    NativeBRepOperation brep_operation = 24;
  }
}

message CommandResult {
  bytes command_id = 1;
  CommandStatus status = 2;
  uint64 committed_revision = 3;
  repeated ResourceInvalidation invalidations = 4;
  optional ErrorDetail error = 5;
}
```

## 8.3 执行状态机

| 状态 | 说明 |
|---|---|
| CREATED | 已创建，尚未进入调度。 |
| PREVIEWING | 前端临时预览，不写命令日志。 |
| PENDING | 后端已接收，等待资源或原生运算。 |
| COMMITTED | 事务提交，revision 增加并发送 invalidation。 |
| FAILED | 保持原 revision，返回结构化错误。 |
| CANCELLED | 任务取消并释放临时资源。 |
| UNDONE / REDONE | 通过新命令实现，不篡改历史记录。 |

## 8.4 并发与版本冲突

- 命令提交必须携带 `base_revision`。
- 本地单用户模式仍需校验 revision，以防异步命令乱序和旧 UI 状态覆盖新状态。
- 可交换的属性命令可自动 rebase；拓扑/B-Rep 命令遇到版本冲突应拒绝并要求重新计算。
- 同一实体或资源的重命令可使用文档级、实体级或资源级锁。

## 8.5 撤销策略

| 类型 | 示例 | 策略 |
|---|---|---|
| 可逆轻命令 | 变换、属性、可见性 | 保存 before/after 或逆命令；O(1) 恢复。 |
| 缓冲区局部修改 | 顶点移动、面属性 | 保存稀疏 patch：entity + accessor + index ranges + old/new bytes。 |
| 拓扑重构 | 合并、拆分、简化 | 保存受影响资源的 copy-on-write 快照或替换块引用。 |
| 精确 B-Rep 操作 | 布尔、倒角、圆角 | 由原生内核生成事务检查点/新 B-Rep 版本；旧版本按引用计数保留。 |
| 导入/批处理 | 新增大量实体 | 整个导入作为事务；撤销移除 manifest 引用，CAS 数据延迟 GC。 |

> **关键约束：** 不得通过深拷贝整棵 Three.js Scene 实现撤销。这会导致 GPU 资源、纹理、TypedArray 和对象引用失控。撤销单位必须是领域命令与不可变资源版本。

## 8.6 事务执行流程

```text
validate(base_revision, permission, entities)
        ↓
prepare temporary resources / native operation
        ↓
write command WAL + inverse/checkpoint references
        ↓
commit SQLite metadata + CAS references atomically
        ↓
increment document revision
        ↓
publish invalidations and command result
```

## 8.7 日志、检查点与恢复

- 每个 COMMITTED 事务先写命令日志/WAL，再更新当前 revision；崩溃后从最近检查点重放。
- 按数据量而非固定命令数生成检查点；命令历史受内存预算约束，旧逆数据可落盘。
- 自动保存与手动保存共享事务机制，但手动保存可生成命名快照/版本标签。
- 恢复时校验命令、资源 hash、项目 schema 和解析器版本，出现不一致则进入只读诊断模式。

## 8.8 命令分类建议

- 文档命令：导入、删除文档、单位/坐标系修改。
- 场景命令：层级、变换、显示属性、图层和材质。
- 网格命令：顶点/边/面 patch、合并、拆分、简化。
- 精确几何命令：B-Rep 布尔、特征和拓扑修改。
- 审阅命令：测量、标注、问题点和选择集。
- 系统命令：迁移、缓存重建、资源修复；默认不进入普通用户撤销栈。
