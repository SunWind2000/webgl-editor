# 2. 技术路线评估与选型

## 2.1 候选路线

| 路线 | 组成 | 优势 | 主要问题 | 结论 |
|---|---|---|---|---|
| A. 全 TypeScript/JS + WASM | Electron/Vue/Three.js；格式尽量由 JS/WASM 解析 | 技术栈统一、迭代快、跨平台 | DWG/STEP 保真、复杂 B-Rep、超大数据与内存风险 | 适合查看器或轻量网格编辑器，不建议作为完整产品主线 |
| B. 混合多进程 | JS 负责 UI/渲染/编排；C++ 侧车负责 CAD/点云重计算 | 性能、隔离、格式能力与开发效率平衡 | 构建、签名、SDK 授权和跨平台 CI 更复杂 | **推荐作为正式产品主线** |
| C. 原生 CAD/渲染内核 + Electron UI | 核心模型和渲染均在原生层，Electron 仅做界面 | 极限性能与内核控制力最高 | 双栈成本高，UI/原生资源同步复杂 | 仅在超大工业模型成为绝对核心时考虑 |

## 2.2 推荐技术栈

| 层 | 推荐 | 说明 |
|---|---|---|
| 桌面壳 | Electron | 主进程只负责生命周期、窗口、菜单、签名更新、受限文件系统和子进程监管。 |
| 前端 | Vue 3 + TypeScript | UI 使用响应式；大型几何缓冲区必须放在响应式系统之外。 |
| 渲染 | Three.js WebGPURenderer + WebGL2 fallback | 以 RendererAdapter 隔离后端差异，WebGL2 作为最低功能基线。 |
| 本地服务 | NestJS + Fastify Adapter | 独立 utilityProcess；HTTP 返回二进制/文件流，SSE 发送任务事件。 |
| 持久化 | SQLite WAL + 内容寻址文件仓库 | SQLite 存元数据、事务与索引；大型几何数据按 chunk 落盘。 |
| STEP/精确几何 | Open CASCADE Technology | STEP/XDE、装配属性、Shape Healing、B-Rep、三角化与精确操作。 |
| DWG/DXF 高保真 | ODA Drawings SDK 或 Autodesk RealDWG | 商业主线二选一；JS DXF 仅作为常用实体快速路径。 |
| 点云 | PCD 解析 + 八叉树/HLOD 转换管线 | 小文件直接加载，大文件转换为分层瓦片并按屏幕误差调度。 |
| 协议 | Protocol Buffers + typed buffer views | 生成 TS/C++ 代码；禁止每个顶点一个 message。 |
| 前端并行 | WebWorker + WASM + SharedArrayBuffer | 统一任务调度、取消、依赖、内存预算和降级。 |

## 2.3 核心选型原则

1. Three.js 是渲染适配层，不是文档数据库，也不是 CAD Kernel。
2. Node.js 是编排与 I/O 层；CPU 密集型几何任务必须进入 Worker、WASM 或原生进程。
3. 精确 B-Rep 与显示网格分离；显示网格可重建，原始/精确数据不可被渲染优化破坏。
4. 协议以分块、可寻址、可增量和版本兼容为第一优先级。
5. 所有跨进程任务均可取消、可超时、可重试，并有明确资源预算。
6. WebGPU 是优化路径而不是唯一运行条件；WebGL2 必须覆盖 P0 核心功能。
7. 优先选择独立原生侧车，而不是将大型 CAD SDK 直接做成 Electron Node-API 插件，以降低 ABI、崩溃和升级耦合。

## 2.4 不推荐的实现方式

- 在 Electron renderer 中开启 `nodeIntegration` 并直接访问文件系统。
- 在 Vue 响应式对象中保存百万级顶点、索引、点云数组或 Three.js 大对象树。
- 使用 JSON 逐点、逐线、逐面传输大型几何数据。
- 使用 SSE 传输 Base64 几何块。
- 通过深拷贝 Three.js Scene 实现撤销重做。
- 让 NestJS 主事件循环直接执行 STEP/DWG 解析、网格布尔或大型点云重建。
