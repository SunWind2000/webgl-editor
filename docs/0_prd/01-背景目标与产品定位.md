# 1. 背景、目标与产品定位

## 1.1 背景

工程领域源数据具有明显异构性：二维图纸以 DXF/DWG 为主，精确三维实体常以 STEP 交换，扫描数据以 PCD 等点云格式存在，实时展示又通常转为 glTF/GLB。单一前端解析器无法同时覆盖格式保真、精确拓扑和超大数据量。

本产品拟提供离线优先的桌面建模与审阅环境，既保留 Web 技术的开发效率，又通过 Electron 获得本地系统能力，并通过原生侧车补齐 CAD 几何能力。

## 1.2 产品目标

- 统一打开、组织、渲染和编辑 DXF、DWG、STEP、PCD、glTF/GLB 等工程数据。
- 在主流工程工作站上保持渐进加载、稳定帧率、可控内存和可取消的后台任务。
- 建立独立于 Three.js 场景树的文档模型、命令事务和版本化图形协议。
- 支持本地项目持久化、崩溃恢复、缓存重建、插件扩展和后续精确建模。
- 支持离线优先运行，并可演进为本地/远程双后端模式。

## 1.3 非目标与边界

- V1 不承诺 AutoCAD、Creo、SolidWorks 全部语义或 100% 无损往返。
- V1 不实现完整参数化草图约束、复杂 NURBS 曲面设计和全量特征树重建。
- 浏览器独立运行不是首要目标；Electron 是主要部署环境。
- 原始 DWG/STEP、精确 B-Rep 与显示代理并存，不强制归一化为单一三角网格。
- P0 阶段以高性能查看、审阅、场景编辑和有限网格编辑为主；精确 CAD 功能分阶段建设。

## 1.4 产品定位

产品定位为面向工程图纸、三维模型和点云的**桌面端高性能编辑与可视化平台**，核心差异化能力包括：

- 多格式统一工程文档模型；
- 精确几何与显示代理双轨存储；
- 大模型渐进加载与资源预算管理；
- 可审计、可恢复的命令事务；
- 面向二次开发的协议和插件能力。

## 1.5 产品成功标准

| 指标 | 定义 |
|---|---|
| 格式可用性 | P0 样本库打开成功率 ≥ 98%，失败必须返回阶段、实体和诊断信息。 |
| 交互性能 | 参考硬件与标准数据档位中，轨道、缩放、选择和剖切达到目标帧率与延迟。 |
| 编辑可靠性 | 连续 1,000 次可撤销命令后状态一致；保存/重启可重放或恢复。 |
| 稳定性 | 导入器或原生侧车崩溃不导致主窗口退出，任务可重试且项目不损坏。 |
| 内存可控 | 达到预算阈值时触发 LOD、卸载与回收，不允许无上限复制 TypedArray。 |
| 可演进性 | 协议、数据模型、渲染后端与原生内核之间存在稳定适配层，避免强耦合。 |
